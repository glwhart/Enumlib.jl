# v0.2 implementation plan

Live execution document for the v0.2 coding effort. Tracks chunks, milestones, status, and surfaced issues. The *design* lives in `research.md` (Phases 1–8); this file is the *execution* companion.

When v0.2 ships, this file is frozen as the v0.2 record. v0.3 gets its own `v0.3-plan.md`.

## Workflow

- **One chunk per session-or-two.** Each chunk has a clear "ends with" milestone and is a single PR-sized change.
- **Sequence:** I propose the design for the chunk, you sanity-check, I write the code with tests, you review the diff, we land it. Each chunk closes with a green test run.
- **Test-driven:** every chunk has tests against existing known counts (`test/runtests.jl` cases for FCC binary at n=4, 8, 12) or hand-verified examples. Tests live in `test/` from chunk 1 onward.
- **Commits:** small, frequent, with descriptive messages. Each chunk's "land" commit references this file (e.g., "Chunk 1: ParentLattice + SymmetryOp — closes v0.2-plan.md §Chunk 1").

## Working agreement (Gus + Claude)

Established during chunk 1 review. Goal: best-quality code with knowledge transfer, not dumbed-down code. Expanded slowly, the human-in-the-loop pace beats the rush-and-debug pace.

- **Best-code-with-teaching, not dumbed-down code.** Claude writes the most idiomatic Julia available. When a construct (`view`, parametric `where`, `hash` pairing, `do`-blocks, broadcasting) might be unfamiliar, Claude flags it inline or in the chunk's review document. Gus reads the explanation; if it lands, that idiom becomes part of the shared vocabulary; if not, Claude rewrites with more pedagogy. We don't trade code quality for legibility — we trade explanation effort.
- **No code is "approved" until both of us understand it.** Gus does a careful 4-eye review on every diff. If something is too clever to follow, Claude teaches it before the code lands. Slow, but correct, and over time the speed compounds as Gus's idiomatic-Julia vocabulary grows.
- **Reviews happen in dedicated docs.** Per chunk, Claude opens a `docs/notes/chunkN-review.md` with numbered items and "Your response:" slots. Gus responds inline; Claude responds to those responses; iterate until everything is locked. Then a single chunk commit lands.
- **Pause before committing.** Claude proposes a chunk's design before writing it; writes the code only after sign-off; then commits. (Chunk 1 went straight from work to commit because we were still calibrating the workflow; chunk 1.1 onward follows this rule.)
- **Comments explain *why*, not just *what*.** A comment like `# wrap into [0,1)` is redundant with the code. A comment like `# Periodic-coordinate gauge fix; matches ASE/pymatgen convention` says why. Claude defaults to the latter.

## Type-system glossary (chunk 1 spillover, reusable for chunks 2+)

Captured during chunk 1 review (see `docs/notes/chunk1-review.md` items 3–5). These terms recur throughout the v0.2 type catalog.

- **Thin wrapper.** A struct that exists to attach a *name* to existing data, with little or no transformation. `SymmetryOp{D}` wraps Spacey's `SpacegroupOp` (same fields: `R::Matrix{Int}`, `t::Vector{Float64}`). The "thin" part means we don't add fields, validate differently, or transform the data — we just rename so we can attach our own methods (without polluting Spacey's namespace) and add the parametric `{D}`.

- **Specialize.** Julia's compiler generates a *separate* compiled version of every function for each concrete type the function is called on. If we write `function apply(op::SymmetryOp{D}, x) where D` and call it with `D=3` and `D=2`, the compiler produces two distinct compiled functions, each with `D` baked in as a literal constant. The compiler can unroll loops of length `D`, omit array-bounds checks, etc. Static dispatch with runtime polymorphism. For 3D crystals (~99% of uses) the compiler treats `D` as a compile-time `3` — code as fast as if we hardcoded "3" everywhere. If we later want 2D, Julia generates a separate fast path automatically.

- **Parametric type / type variable / `where`.** `where D` introduces a type variable — like a generic parameter in Java/C++/TypeScript. Reading a signature `f(a::SymmetryOp{D}, b::SymmetryOp{D}) where D`: both arguments must be `SymmetryOp` parametric on the *same* `D`. The dispatcher binds `D` per call. Without `where D`, comparing `SymmetryOp{3}` to `SymmetryOp{2}` would type-check but probably crash inside; with `where D`, mismatched-D calls fall through to the default method (which usually returns `false` or throws `MethodError`). The single-`D` form expresses "same dimension" at the type level for free.

- **Pairing rule (`==` and `hash`).** Any pair `(a, b)` where `a == b` MUST satisfy `hash(a) == hash(b)`. Otherwise hash-based collections (`Dict`, `Set`, `unique`) misbehave: equal values get sent to different buckets. Julia's default `hash` for a custom struct hashes the struct's *identity* (memory-like), not its contents — so two structurally-equal struct instances get *different* default hashes unless we override. Discipline: define `==` and `hash` together, both reading the same fields. The pattern `hash(op.R, hash(op.t, h))` chains hashes through the fields, mixing each into the seed `h` produced by the previous one.

- **`view` vs slice.** `view(A, :, j)` returns a *reference* into `A` (no copy); `A[:, j]` returns a *copy*. For read-only operations like computing a column norm, `view` is both faster (no allocation) and the idiomatic signal that we're not mutating. Use `view` whenever you don't need ownership of the data.

- **Hadamard ratio.** A scale-free measure of basis-matrix non-singularity: `|det(A)| / prod(‖aⱼ‖)` over the columns of `A`. Equals `1` for orthogonal columns, `0` for linearly dependent columns. Dimensionless — works regardless of unit choice. Used in `ParentLattice{D}` to detect numerically singular bases. Equivalent (inverse) of `MinkowskiReduction.orthogonalityDefect`.
